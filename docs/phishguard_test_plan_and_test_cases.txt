PhishGuard Test Plan and Test Cases
===================================

Document Purpose
----------------
This document provides a complete, project-ready test plan and test case set for PhishGuard (Extension + Backend), including:
- Test steps
- Test data
- Expected output
- Actual result (to be filled during execution)
- Pass/Fail (to be filled during execution)


1) Test Plan
------------

1.1 Objective
- Validate that PhishGuard correctly detects phishing/safe emails using multi-signal analysis.
- Validate integration between extension UI, Gmail context extraction, backend analysis, VT URL/file-hash checks, and action buttons.
- Validate user-facing explainability, risky-link rendering, and spam-action flow.

1.2 Scope
In Scope:
- Gmail content extraction in extension
- Backend `/analyze-email` risk evaluation
- Attachment analyzer + VT file-hash checks
- URL/link risk pipeline (existing tuned behavior)
- Popup action buttons (Mark Safe, Report Phishing, Learn More)
- Local education page rendering
- Feedback endpoint logging

Out of Scope:
- Model retraining automation from feedback
- Non-Gmail mailbox providers
- Enterprise SSO and multi-tenant backend auth

1.3 Test Types
- Functional Testing
- Integration Testing
- API Testing
- UI/UX Testing
- Negative Testing
- Security/Privacy Validation
- Compatibility Testing (Chrome + Firefox)
- Performance/Responsiveness (basic practical checks)

1.4 Test Environment
- OS: Windows 10/11
- Browser: Chrome (latest), Firefox (latest)
- Extension loaded in developer mode
- Backend API running locally (or deployed endpoint configured in extension)
- VT API key configured in backend environment
- Test Gmail account with sample emails

1.5 Entry Criteria
- Extension builds/loads without manifest errors
- Backend server starts successfully
- OAuth configuration completed and consent screen/test user configured
- Network connectivity available for Gmail API and VT calls

1.6 Exit Criteria
- All Critical and High-priority tests pass
- No blocker defects in analysis flow, report-phishing action, or core verdict display
- Demo checklist scenarios execute successfully


2) Test Data Set
----------------

TD-01 Safe Newsletter Email
- Sender: trusted domain newsletter
- Body: informational text, normal CTA links
- Links: known benign
- Attachments: none

TD-02 Credential Phish Email
- Sender spoof-like display name
- Body: urgent account lock warning
- Links: lookalike domain (e.g., lnstagram.com style)
- Attachments: none

TD-03 Malware Attachment Email
- Sender: unknown external
- Body: invoice/payment urgency
- Links: optional
- Attachment: suspicious filename/ext (e.g., Invoice_2026.exe or macro-enabled doc equivalent for test)
- Hash reputation: known suspicious/malicious where possible

TD-04 Safe Attachment Email
- Sender: trusted contact
- Body: normal communication
- Attachment: benign file (PDF/JPG)
- Hash reputation: unknown or benign

TD-05 Mixed Signal Email
- Sender: somewhat suspicious
- Body: social engineering language
- Links: one benign + one suspicious
- Attachment: neutral

TD-06 Plain-text URL Email
- Body contains risky URL as text (not anchor tag)
- Used to validate plain-text URL extraction consistency

TD-07 Noisy/Minimal Content Email
- Very short content with little context
- No links, no attachments

TD-08 API Failure Simulation
- Backend running without VT key or with forced VT timeout
- Used to verify graceful fallback behavior
